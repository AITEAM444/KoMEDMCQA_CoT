# 이 연구가 하려는 일 — 쉽게 읽는 개요

> 한 줄 요약: **"답은 맞았지만 사실은 대충 둘러댄 풀이"를 골라내면, 더 적은 데이터로도 AI를 더 잘 가르칠 수 있다**는 걸 증명하려는 연구입니다.

---

## 1. 큰 그림 (비유로 먼저)

의대 시험문제(객관식)를 잘 푸는 **작은 AI(학생)**를 만들고 싶습니다.
그래서 아주 똑똑한 **큰 AI(선생, DeepSeek-R1)**에게 문제를 풀게 하고, 그 **풀이 과정(생각의 흐름, CoT)**을 교재로 삼아 학생 AI에게 베껴 학습시킵니다. 이걸 **distillation(증류)** 이라고 합니다.

그런데 선생의 풀이 중에는 **나쁜 교재**가 섞여 있습니다:

> 정답은 C로 맞췄는데, 사실 **A를 정답이라고 우겨도 똑같이 그럴듯하게** 설명을 지어낼 수 있는 풀이.

이건 진짜로 추론한 게 아니라 **답을 정해놓고 끼워 맞춘 변명(사후합리화, post-hoc rationalization)** 입니다. 이런 걸로 학생을 가르치면 "그럴듯하게 말하는 법"만 배우지 "제대로 추론하는 법"은 못 배웁니다.

**이 연구의 핵심 아이디어:**
풀이가 진짜인지 가짜인지 알아보려면, **일부러 답을 오답으로 바꿔서** 다시 설명을 시켜봅니다.
- **건강한 풀이**: 오답을 정당화하라고 하면 잘 못 함 → 원래 답일 때와 **설명 강도 차이(gap)가 큼**.
- **사후합리화 풀이**: 아무 답이나 똑같이 잘 둘러댐 → **gap이 작음** → **탈락**.

이 필터를 **반사실 충실성 필터(Inverse-MATCHA)** 라고 부릅니다. ("반사실 = counterfactual = 만약 답이 달랐다면?")

---

## 2. 무엇을 증명하려는가 (주장)

흔히 쓰는 **"범용 LLM 채점기(judge)"** 는 "이 풀이 잘 썼나?"를 점수 매깁니다. 그런데 **정답을 맞춘 풀이들 사이에서는** 이 채점기가 변별력을 잃습니다 — 다 그럴듯해 보이거든요.

반면 반사실 프로빙은 **"답을 바꿨을 때 설명이 무너지나?"** 라는 **완전히 다른 축**에서 사후합리화를 잡아냅니다.

> **증거의 형태:** 반사실 필터로 거른 데이터로 학생을 가르치면, **데이터가 더 적은데도** 정확도가 유지되거나 올라간다.

---

## 3. 비교군 5개 (arm) — 공정한 실험을 위한 장치

같은 선생 풀이 더미에서 출발해, **거르는 방법만 다르게** 한 5개 묶음을 만들고 각각으로 학생을 학습시켜 시험 점수를 비교합니다.

| 묶음 | 무엇 | 왜 필요한가 |
|---|---|---|
| **C0** | 안 거름 (원본 전체) | 바닥선(하한). 필터가 아예 없으면? |
| **C1** | 기본 정제 (정답·가독성·형식·길이) | 누구나 하는 상식적 청소 |
| **C-rand** | C1에서 **무작위로** \|C3\|개만 남김 | **크기 통제군.** "그냥 양이 줄어서 좋아진 거 아냐?"를 반박 |
| **C2** | C1 + **범용 채점기** 상위 \|C3\|개 | **대조군.** "일반 채점기로는 못 잡는다"를 보임 |
| **C3** | C1 + **반사실 필터** (이 연구의 제안) | **제안 방법 (주인공)** |

**핵심:** C-rand와 C2는 모두 **C3와 똑같은 개수**로 맞춥니다. 크기가 같아야 "방법의 차이"만 비교할 수 있기 때문입니다. → 그래서 **C3 개수(|C3|)를 먼저 알아야** C2·C-rand를 만들 수 있습니다.

### 검증할 가설
- **H2:** C3 ≥ C1 — 제안 필터가 기본 정제보다 같거나 낫다.
- **H2 통제:** C3 > C-rand — 이득이 단순 "데이터 축소" 때문이 아니다.
- **H3:** C3 > C2 (같은 크기) — **일반 채점기로는 못 잡는 걸** 반사실이 잡는다. ← 가장 중요한 주장.

---

## 4. 두 종류의 채점기 — 헷갈리지 말 것

| | **C2 채점기** (범용) | **C3 채점기** (반사실, 제안) |
|---|---|---|
| 무엇을 보나 | 풀이 글 자체의 품질 (**답 안 바꿈**) | 답을 **오답으로 바꿔도** 똑같이 정당화되나 |
| 점수 | `c2_score` (1~5) | `gap = orig_score − max(cf_score)` |
| 판정 | 점수 **상위 \|C3\|개**를 C2로 | gap이 **작으면 탈락** (사후합리화) |

**신호(점수) 용어:**
- `orig_score` (1~5): 원래 풀이가 **정답**을 정당화하는 강도.
- `cf_score` (1~5): 일부러 만든 풀이가 **오답**을 정당화하는 강도.
- `gap = orig_score − max(cf_score)`: **클수록 건강**, 작을수록 사후합리화 의심.
- `c2_score` (1~5): 범용 품질 점수 (C2 선별용).

## 5. 데이터는 한 파일에 쌓인다

모든 판정 결과는 **`data/unified.jsonl` 한 파일**에 누적됩니다. 한 줄(문제 하나)이 이렇게 생겼습니다:

```json
{
  "id": "dentist_0000",
  "question": "...", "choices": ["...", ...], "gold": 2,
  "think": "선생 AI의 풀이 과정 ...",
  "filters": { "C0": true, "C1": true, "C2": true, "C3": false, "C-rand": false },
  "signals": { "c2_score": 4.0, "counterfactual": { "orig_score": 5, "gap": 2.3 } }
}
```

- `filters`: 이 문제가 어느 묶음에 속하는지 (한 줄에 5개 묶음 동시 표시).
- `signals`: 채점 점수들.

학습할 때 비로소 "C3=true인 줄만" 하는 식으로 잘라 씁니다. **묶음을 따로 파일로 안 쪼개고 한 파일에 다 모으는 이유** = 리포트가 한 번에 "C1 몇 건, C3 몇 건"을 셀 수 있어서.

## 6. 전체 파이프라인 (단계별)

```
1) 선생 풀이 생성   → train_cot_fewshot.jsonl
2) C0/C1 판정       → unified.jsonl (C0/C1 채움)
3) 반사실 가지(C3):
   3a) 오답 풀이 생성    → train_cf.jsonl
   3b) 채점(precompute)  → cf_judged.json   (★비싼 API 단계)
   3c) merge-c3          → unified 에 C3 채움  ← 여기서 |C3| 확정
4) 범용 가지(C2):
   4a) judge_general    → unified 에 c2_score 채움
   4b) merge-c2         → c2_score 상위 |C3|개 = C2
5) C-rand              → 무작위 |C3|개 = C-rand
6) 학습 (arm × seed)
7) 평가 (test 점수)
8) 통계 (평균±표준편차, 신뢰구간, McNemar 검정)
9) 리포트 (report.md)
```

> ⚠️ **순서 주의:** C3(3c)가 |C3| 개수를 정하므로 **반드시 C2(4b)·C-rand(5)보다 먼저** 끝나야 합니다.
> `c2_score`(4a)는 미리 계산해 둬도 되지만, **상위 몇 개를 자를지(4b)는 |C3|를 안 뒤**에만 가능합니다.

### 6~9단계 자세히 (학습 → 평가 → 통계 → 리포트)

여기서부터는 더 이상 데이터를 거르는 단계가 아니라, **거른 데이터로 학생을 실제로 가르치고 점수를 매겨 가설을 검증**하는 단계입니다.

**6) 학습 — `arm × seed` 묶음마다 LoRA 어댑터 1개씩**

- 5개 묶음(C0·C1·C2·C3·C-rand) × 3개 seed(42·43·44) = **총 15번의 독립 학습**.
- 코드는 `for arm: for seed:` **이중 루프로 한 번에 하나씩 순차** 학습합니다. GPU에는 항상 **베이스 모델(Qwen3-8B) + LoRA 어댑터 1개**만 올라가므로, 15개가 동시에 메모리에 올라가는 일은 **없습니다.** → 한 번 학습할 만큼의 GPU 메모리만 있으면 됩니다.
- 결과물: 묶음·seed별 어댑터 폴더 (예: `output/qwen3-8b-c3-s42`).

> **Q. seed랑 arm을 나눠서 따로 돌려도 되나? → 됩니다.**
> 각 `(arm, seed)` 학습은 서로 완전히 독립이고 결과 폴더 이름이 다르므로, **한 명령에 다 넣든 여러 명령으로 쪼개든 결과는 동일**합니다.
> - GPU 한 대로 한꺼번에 돌리기 부담되면 → seed별/arm별로 **명령을 나눠** 며칠에 걸쳐 돌려도 됩니다.
> - GPU가 여러 대면 → 묶음을 나눠 **병렬**로 돌릴 수도 있습니다 (다른 GPU에 다른 arm/seed 배정).

> **Q. 왜 seed를 쓰나? (중요)**
> 1. **재현성** — seed를 고정하면 데이터 셔플 순서·LoRA 가중치 초기화·dropout이 매번 똑같아져, **누가 돌려도 같은 결과**가 나옵니다 (논문 재현성 요건).
> 2. **분산(노이즈) 추정** — 학습에는 무작위성이 있어 한 번 돌린 점수는 운에 따라 흔들립니다. seed 3개로 돌려 **평균 ± 표준편차**를 내야 "C3가 C1보다 높다"가 **진짜 효과인지 그냥 운인지** 구분할 수 있습니다. ← 8단계 신뢰구간·검정이 바로 이걸 위한 것.
> 3. **공정 비교** — 모든 묶음을 **같은 seed들**로 학습하면 무작위 요인이 통제되어, 점수 차이가 "운"이 아니라 "데이터(묶음) 차이"에서 왔다고 말할 수 있습니다.

**7) 평가 — 학습된 어댑터마다 test셋을 풀려 채점**

- 어댑터 1개당 test 문제를 전부 풀게 하고, 문제별 정답 여부(`correct`)를 줄 단위로 저장합니다 (예: `results/eval_C3_s42.jsonl`).
- 15개 어댑터 → 15개 결과 파일. 평가도 어댑터를 **하나씩 불러** 돌리므로 GPU 부담은 학습과 동일하게 1개분뿐입니다.

**8) 통계 — seed들을 모아 "차이가 유의미한가"를 판정**

- **묶음 내**: 같은 묶음의 seed 결과들(예: `eval_C3_s*.jsonl`)을 모아 **평균 ± 표준편차**와 **95% 신뢰구간**을 냅니다.
- **묶음 간 (McNemar 검정)**: 두 묶음이 **같은 문제에서** 누가 맞고 틀렸는지를 짝지어 비교합니다 ([stats.py](src/eval/stats.py#L31-L43)). "A는 맞았는데 B는 틀린 수(b)"와 "그 반대(c)"만 가지고 p-value를 내므로, 단순 평균 비교보다 **차이가 우연인지**를 더 엄밀하게 봅니다.
- 출력 `stats.json`에 H2(C3≥C1)·H2통제(C3>C-rand)·H3(C3>C2) 가설 판정이 들어갑니다.

**9) 리포트 — 숫자를 사람이 읽는 한 장으로**

- 위에서 나온 묶음 크기·신호 분포·정확도·검정 결과를 `report.md` 하나로 묶고, 각 가설에 ✅/❌를 붙입니다 (자세한 항목은 아래 8장 참고).

## 7. 실행 코드

> 이 저장소(Windows)에서는 `python` 대신 `py`를 씁니다. API 키 필요:
> `DEEPSEEK_API_KEY`(생성), `OPENAI_API_KEY`(채점).

### (A) 한 방에 전체 자동
```bash
bash scripts/run_pipeline.sh   # 산출물이 있으면 그 단계는 건너뜀
```

### (B) 단계별 수동 실행

```powershell
# 1) 선생(R1) 풀이 생성
py src/generation/generate_traces.py `
   --model deepseek-r1 --total -1 --split train --prompt-mode fewshot `
   --workers 16 --output data/train_cot_fewshot.jsonl

# 2) C0/C1 판정 → unified 생성
py src/dataset/build_arms.py unified `
   --input data/train_cot_fewshot.jsonl `
   --output data/filtered/unified.jsonl

# 3a) 반사실(오답) 풀이 생성
py src/filters/counterfactual_adapter.py `
   --total -1 --split train --workers 16 `
   --output data/counterfactual/train_cf.jsonl

# 3b) 반사실 채점 (★비싼 단계 — OpenAI judge)
py -m filters.counterfactual.precompute `
   --input data/counterfactual/train_cf.jsonl `
   --output data/cf_judged.json --workers 16

# 3c) merge-c3 → filters.C3 채움 (|C3| 확정)
py src/dataset/build_arms.py merge-c3 `
   --unified data/filtered/unified.jsonl `
   --cf data/cf_judged.json `
   --output data/filtered/unified.jsonl

# 4a) 범용 judge 채점 → c2_score 채움
py src/filters/judge_general.py `
   --unified data/filtered/unified.jsonl `
   --output data/filtered/unified.jsonl --workers 16

# 4b) merge-c2 → c2_score 상위 |C3|개 = C2
py src/dataset/build_arms.py merge-c2 `
   --unified data/filtered/unified.jsonl `
   --output data/filtered/unified.jsonl

# 5) C-rand → 무작위 |C3|개
py src/dataset/build_arms.py make-crand `
   --unified data/filtered/unified.jsonl `
   --output data/filtered/unified.jsonl --seed 42
   

# 6) 학습 (arm × seed) — 5묶음 × 3seed = 15개 어댑터를 "순차로" 학습
#    GPU엔 늘 베이스+LoRA 1개만 올라감 → 한꺼번에 돌려도 메모리 OK
py src/train/train_lora.py `
   --unified data/filtered/unified.jsonl `
   --arms C0 C1 C2 C3 C-rand --seeds 42 43 44

# 6-별법) 부담되면 묶음/seed를 나눠 따로 돌려도 결과 동일 (어댑터 폴더가 분리됨)
#   예: C3만 seed 하나씩
py src/train/train_lora.py --unified data/filtered/unified.jsonl --arms C3 --seeds 42
py src/train/train_lora.py --unified data/filtered/unified.jsonl --arms C3 --seeds 43
#   예: GPU 여러 대면 묶음을 나눠 병렬 (각 명령을 다른 GPU에서)
#   $env:CUDA_VISIBLE_DEVICES=0; py ... --arms C0 C1 --seeds 42 43 44
#   $env:CUDA_VISIBLE_DEVICES=1; py ... --arms C2 C3 C-rand --seeds 42 43 44

# 7) 평가 (test) — arm·seed 마다
py src/eval/evaluate.py --model Qwen/Qwen3-8B `
   --lora output/qwen3-8b-c3-s42 `
   --split test --output results/eval_C3_s42.jsonl

# 8) 통계
py src/eval/stats.py `
   --arm C3 results/eval_C3_s*.jsonl `
   --arm C1 results/eval_C1_s*.jsonl `
   --mcnemar C3 C1 --mcnemar C3 C-rand --mcnemar C3 C2 `
   --output results/stats.json

# 9) 리포트
py scripts/make_report.py `
   --unified data/filtered/unified.jsonl `
   --stats results/stats.json `
   --output results/report.md
```

> 💡 이미 `judge_general`을 먼저 돌려 `c2_score`가 든 파일(`after_c2.jsonl`)이 있다면,
> 4a를 다시 할 필요 없이 **그 파일을 3c의 `--unified` 입력으로** 주면 c2_score가 그대로 따라옵니다.

## 7-2. 사전 학습 어댑터 불러와서 평가하기 (HuggingFace)

학습된 LoRA 어댑터가 이미 [AIteam4/KoMEDMCQA-LoRA](https://huggingface.co/AIteam4/KoMEDMCQA-LoRA) 에 올라와 있습니다.
직접 학습(6단계)을 건너뛰고 바로 평가(7단계)부터 시작할 수 있습니다.

**어댑터 이름 규칙:** `qwen3-8b-<arm>-s<seed>` (예: `qwen3-8b-c3-s42`)

```powershell
# 1) 원하는 어댑터 다운로드 (예: C3, seed 42)
hf download AIteam4/KoMEDMCQA-LoRA --repo-type model `
  --include "qwen3-8b-c3-s42/*" --local-dir output

# 전체(2 GB) 한꺼번에 받으려면:
# hf download AIteam4/KoMEDMCQA-LoRA --repo-type model --local-dir output

# 2) 평가 실행 (7단계와 동일)
py src/eval/evaluate.py --model Qwen/Qwen3-8B `
   --lora output/qwen3-8b-c3-s42 `
   --split test --output results/eval_C3_s42.jsonl
```

모든 arm·seed를 한꺼번에 평가하려면:
```powershell
hf download AIteam4/KoMEDMCQA-LoRA --repo-type model --local-dir output

foreach ($arm in @("c0","c1")) {
  foreach ($s in @(42,43,44)) {
    py src/eval/evaluate.py --model Qwen/Qwen3-8B `
       --lora "output/qwen3-8b-$arm-s$s" `
```

## 8. 리포트는 무엇을 보여주나
