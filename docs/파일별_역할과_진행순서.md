# 파일별 역할 · 작동방법 · 입력/출력 — 그리고 진행 순서

> 이 문서는 **"어떤 파일이 무엇을 받아서(input) 무슨 일을 하고(작동) 무엇을 내놓는지(output)"** 를
> 코드를 안 열어봐도 알 수 있게 풀어 쓴 설명서입니다.
> 연구의 큰 그림(왜 이걸 하는가)은 [연구개요.md](연구개요.md) 를 먼저 읽으면 좋습니다.
> 여기서는 **"실제로 무엇이 돌아가는가"** 에 집중합니다.

---

## 0. 30초 요약 — 전체가 한눈에

```
[선생 AI가 문제 풀이] → [기본 청소] → [가짜 추론 골라내기] → [학생 AI 학습] → [시험] → [점수 비교]
   generate_traces      build_arms      counterfactual         train_lora     evaluate    stats
        ↓                  ↓               ↓                       ↓             ↓          ↓
  train_cot.jsonl     unified.jsonl   unified.jsonl(C3 채움)   LoRA 폴더    eval_*.jsonl  stats.json
```

- 모든 중간 결과는 **`data/unified.jsonl` 한 파일**에 차곡차곡 쌓입니다 (별도 파일로 안 쪼갬).
- 데이터를 만드는 데는 **두 종류의 외부 AI** 를 빌립니다:
  - **선생(Teacher) = DeepSeek-R1** → 문제 풀이(추론)를 생성. `DEEPSEEK_API_KEY` 필요.
  - **채점관(Judge) = GPT-5** → 풀이가 얼마나 그럴듯한지 1~5점 채점. `OPENAI_API_KEY` 필요.
- 학습되는 **학생(Student) = Qwen3-8B** (내 GPU에서 LoRA 방식으로 학습).

## 1. 핵심 용어 5개만 (이것만 알면 됨)

| 용어 | 쉬운 뜻 |
|---|---|
| **CoT (Chain-of-Thought)** | AI가 답에 이르기까지 쓴 "풀이 과정 / 생각의 흐름" 글. |
| **distillation (증류)** | 똑똑한 선생 AI의 풀이를 교재 삼아 작은 학생 AI에게 베껴 학습시키는 것. |
| **반사실(counterfactual)** | "만약 답이 (오답으로) 달랐다면?" 하고 일부러 답을 바꿔 다시 풀게 해보는 것. |
| **gap (격차)** | `정답을 정당화한 점수 − 오답을 정당화한 점수`. **클수록 건강한 풀이**, 작으면 "아무 답이나 둘러대는" 가짜 추론. |
| **arm (비교군)** | 같은 풀이 더미를 **다른 방법으로 걸러낸** 5개 묶음(C0/C1/C2/C3/C-rand). 공정 비교용. |

---

## 2. 데이터가 흘러가는 길 (파일 → 파일)

각 단계가 **무슨 파일을 읽어서 무슨 파일을 만드는지**를 그림으로:

```
KorMedMCQA (huggingface 데이터셋, 한국 의료 객관식)
   │
   │ ① generate_traces.py  (선생 R1이 train 문제를 품)
   ▼
data/train_cot_fewshot.jsonl   ← 문항별 풀이(reasoning) + 선생이 고른 답
   │
   │ ② build_arms.py unified  (문제·선지·정답 합치고 기본필터 C0/C1 판정)
   ▼
data/unified.jsonl   ← 여기서부터 모든 결과가 이 한 파일에 누적됨
   │
   ├─ ③ 반사실(C3) 가지 ───────────────────────────────────┐
   │   counterfactual_adapter.py → train_cf.jsonl           │ (오답 정당화 풀이 생성)
   │   precompute.py             → cf_judged.json           │ (원본·오답 풀이 채점 ★비쌈)
   │   build_arms.py merge-c3    → unified.jsonl 에 C3 채움  │ ← 여기서 |C3| 개수 확정!
   │                                                         ▼
   ├─ ④ 범용 judge(C2) 가지                             unified.jsonl
   │   judge_general.py       → unified 에 c2_score 채움
   │   build_arms.py merge-c2 → c2_score 상위 |C3|개 = C2
   │
   ├─ ⑤ C-rand (무작위 |C3|개)
   │   build_arms.py make-crand → unified 에 C-rand 채움
   ▼
data/unified.jsonl   ← 5개 묶음(C0~C-rand) 플래그가 전부 채워진 최종본
   │
   │ ⑥ train_lora.py  (묶음별 데이터를 잘라 학생 AI를 LoRA로 학습)
   ▼
output/qwen3-8b-<arm>-s<seed>/   ← 학습된 LoRA 어댑터 (묶음×시드마다 1개)
   │
   │ ⑦ evaluate.py  (test 문제로 시험)
   ▼
results/eval_<arm>_s<seed>.jsonl   ← 문항별 정답/오답 기록
   │
   │ ⑧ stats.py  (시드 평균·신뢰구간·검정) → ⑨ make_report.py
   ▼
results/stats.json → results/report.md   ← 가설 ✅/❌ 최종 리포트
```

# 3. 파일별 상세 설명 (진행 순서대로)

각 파일을 **"역할 / 어떻게 작동 / 입력 / 출력 / 실행 예시"** 로 정리했습니다.

---

## ① `src/generation/generate_traces.py` — 선생 AI에게 문제 풀게 하기

**역할:** 한국 의료 객관식(KorMedMCQA)의 **train 문제 3,401개**를 선생 AI(DeepSeek-R1)에게
풀게 하고, 그 **풀이 과정(reasoning)** 을 모아 파일로 저장합니다. 이게 나중에 학생을 가르칠 "교재 원본"입니다.

**어떻게 작동하나:**
1. huggingface에서 `sean0042/KorMedMCQA` 데이터(의사/간호사/약사/치과의사 4분야)를 불러옵니다.
2. 각 문제에 대해 **영어 지시문 + 그 분야 한국어 예시 5개(fewshot)** 를 붙여 R1에게 보냅니다.
   - 추론 언어를 강제하지 않습니다(R1이 자연스럽게 한국어/영어로 생각하게 둠).
3. R1은 답뿐 아니라 **`reasoning_content`(진짜 내부 생각)** 을 따로 돌려주는데, 이걸 핵심 신호로 잡습니다.
4. 동시에 여러 문제를 병렬 호출(`--workers 16`)하고, 중간에 끊겨도 **이어서 재개(체크포인트)** 됩니다.

**입력:**
- huggingface KorMedMCQA `train` split (자동 다운로드)
- API 키 `DEEPSEEK_API_KEY`

**출력:** `data/train_cot_fewshot.jsonl` — 한 줄 = 문제 하나
```json
{ "subset": "doctor", "sample_idx": 12, "predicted": "C",
  "reasoning_content": "환자의 증상은 ... 따라서 C가 적절하다",
  "reasoning": "정답: C", "is_correct": true, "think_lang": "korean", ... }
```

**실행 예시:**
```powershell
py src/generation/generate_traces.py --model deepseek-r1 --total -1 --split train `
   --prompt-mode fewshot --workers 16 --output data/train_cot_fewshot.jsonl
```

> ⚠️ **절대 규칙:** `test` split 으로는 절대 생성하지 않습니다. test는 맨 마지막 시험(평가) 전용입니다.

## ② `src/dataset/build_arms.py` — 데이터 통합 + 묶음(arm) 만들기

이 파일은 **명령(서브커맨드)이 5개**라서, 파이프라인 여러 군데에서 계속 등장합니다.
공통점: **`data/unified.jsonl` 한 파일을 읽어서 표시(flag)를 더 채운 뒤 다시 저장**합니다.

### ②-a `unified` — 통합 스키마 만들기 + 기본필터(C0/C1) 판정

**역할:** ①번 풀이 파일과 원본 문제(질문·선지·정답)를 합치고, **누구나 하는 기본 청소(C1)** 를 통과했는지 판정합니다.

**어떻게 작동하나:**
- 풀이 파일의 각 줄을 원본 문제와 `(분야, 문항번호)` 로 짝지어 질문·선지·정답을 붙입니다.
- [src/filters/standard.py](../src/filters/standard.py) 의 C1 4개 검사를 돌려 통과 여부를 기록합니다.
- `C0`(=원본 전체)는 항상 True, `C1`은 4개 다 통과해야 True, `C2`/`C3`는 일단 비워둠(None).

**입력:** `data/train_cot_fewshot.jsonl` (①의 출력)
**출력:** `data/unified.jsonl` — **앞으로 모든 결과가 쌓일 중심 파일**
```json
{ "id": "doctor_0012", "subject": "doctor", "question": "...", "choices": ["...",...],
  "gold": 2, "think": "환자의 증상은 ...", "final_answer": "C",
  "filters": { "C0": true, "C1": true, "C2": null, "C3": null },
  "signals": { "c1": {...}, "think_lang": "korean" } }
```

### ②-b `merge-c3` — 반사실 채점결과를 합쳐 C3 판정 (→ ③에서 자세히)
### ②-c `merge-c2` — 범용 judge 점수 상위 |C3|개를 C2로 (→ ④에서 자세히)
### ②-d `make-crand` — C1 통과분에서 무작위 |C3|개를 C-rand로 (→ ⑤에서 자세히)

### ②-e `export` — 한 묶음을 학습용 형식으로 잘라내기

**역할:** `unified.jsonl` 에서 특정 묶음(예: C3=true)만 골라, 학습기가 먹는 **대화 형식(messages)** 으로 변환합니다.
**입력:** `unified.jsonl` + 묶음 이름(`--arm C3`)
**출력:** `data/sft/train_C3.jsonl`
```json
{ "messages": [
    {"role": "user", "content": "다음 문제를 풀이하세요...\n[문제]...\n[선택지]..."},
    {"role": "assistant", "content": "환자의 증상은 ...\n정답: C"} ] }
```
> 보통 직접 안 돌립니다 — 학습 스크립트(`train_lora.py`)가 내부에서 자동 호출합니다.

**실행 예시(통합·C0/C1 판정):**
```powershell
py src/dataset/build_arms.py unified --input data/train_cot_fewshot.jsonl --output data/unified.jsonl
```

## ②보조 `src/filters/standard.py` — 기본 청소(C1)의 실제 검사 규칙

**역할:** "누구나 하는 상식적 데이터 청소" 4가지를 코드로 구현. `build_arms unified` 가 불러 씁니다.
직접 실행하는 파일은 아니고, **규칙 모음**입니다.

| 검사 | 무엇을 보나 | 탈락 기준(기본값) |
|---|---|---|
| ① correctness | 선생이 고른 답이 정답인가 | 틀리면 탈락 |
| ② readability | 글이 안 깨졌나 (이상문자·코드블록·끊김) | 이상문자 비율 >15% 등 |
| ③ format_parse | 답을 뽑아낼 수 있고 추론이 존재하나 | 답 파싱 실패 시 탈락 |
| ④ length | 추론 길이가 적당한가 | 단어 <20(너무 짧음) 또는 >4000(발산) |

**임계값 위치:** [configs/pipeline_config.yaml](../configs/pipeline_config.yaml) 의 `c1_filters`.

## ③ 반사실 필터(C3) — 이 연구의 핵심 (제안 방법)

C3는 **3개 파일이 순서대로** 협력합니다. 핵심 아이디어는:
> "정답을 정당화한 점수"와 "**일부러 오답을 정당화시킨** 점수"를 비교해서,
> **차이(gap)가 작으면 = 아무 답이나 둘러대는 가짜 추론** 이므로 버린다.

### ③-1 `src/filters/counterfactual_adapter.py` — 오답을 정당화하는 풀이 생성

**역할:** 각 문제에서 **오답 1개를 골라 "이게 정답이다"** 라고 우기며 그것을 정당화하는 풀이를 R1에게 새로 쓰게 합니다.

**어떻게 작동하나:**
- ①번과 **똑같은 생성 인프라**(영어 지시 + 한국어 fewshot, R1 reasoning_content 캡처)를 재사용합니다.
- 다른 점: 지시문이 "정답은 {오답글자}다. 왜 맞는지 정당화하라"로 바뀝니다.
- 어떤 오답을 고를지는 `(seed + 문항번호)` 로 정해져 **언제 돌려도 같은 오답** 이 재현됩니다.

**입력:** KorMedMCQA `train` (자동) + `DEEPSEEK_API_KEY`
**출력:** `data/counterfactual/train_cf.jsonl`
```json
{ "subset": "doctor", "sample_idx": 12, "gold_letter": "C", "wrong_letter": "A",
  "cf_reasoning_content": "A가 정답인 이유는 ...", "predicted": "A", ... }
```

### ③-2 `filters/counterfactual/precompute.py` — 채점 (★가장 비싼 단계)

**역할:** 원본 풀이와 오답 풀이를 **채점관(GPT-5)** 에게 1~5점으로 매기게 합니다.
**모든 비싼 API 호출이 여기서 끝납니다** (다음 단계는 산수만).

**어떻게 작동하나:**
- **원본 풀이** → "정답(gold)을 얼마나 강하게 정당화했나" 채점 → `orig_score`
- **오답 풀이** → "그 오답을 얼마나 강하게 정당화했나" 채점 → `cf_score`
- 채점관은 [filters/counterfactual/judge.py](../filters/counterfactual/judge.py) 의 `_CF_RUBRIC` 규칙을 씁니다 (③보조 참고).
- 채점 결과를 각 샘플의 `metadata.counterfactual` 에 박아둡니다. 중간 끊김 대비 체크포인트(`.ckpt.jsonl`) 내장.

**입력:** `data/counterfactual/train_cf.jsonl` (③-1 출력) + `OPENAI_API_KEY`
**출력:** `data/counterfactual/cf_judged.json`
```json
{ "id": "doctor_0012", "metadata": { "counterfactual": {
    "orig_score": 5.0,
    "counterfactuals": [ {"target_letter":"A", "cf_score": 2.0, "cf_cot":"..."} ] } } }
```

### ③-3 `filters/counterfactual/run.py` — 통과/탈락 판정 (산수만, 무료)

**역할:** 박아둔 점수로 **gap을 계산**해서 통과/탈락을 정합니다. API 호출이 없어서 임계값을 바꿔 몇 번을 돌려도 비용 0.

**판정 규칙:**
- `gap = orig_score − max(cf_score)`
- `gap < 1.5` (기본) → "어떤 답이든 비슷하게 정당화" = **사후합리화 의심 → 탈락**
- 추가 안전망: `orig_score < 2.0` (원본조차 정당화 실패) → 무조건 탈락
- 보조 신호 `hedge_gap`(망설임 표지 차이)도 계산하지만 기본은 참고용.

> 이 파일은 단독 실행도 되지만, 보통 **`build_arms.py merge-c3` 가 이 판정 로직을 불러서** 사용합니다.

### ③-merge `build_arms.py merge-c3` — C3 플래그 채우기 (★|C3| 개수 확정)

**역할:** ③-2 채점결과 + ③-3 판정 로직을 합쳐, `unified.jsonl` 의 각 줄에 `filters.C3` 를 채웁니다.
- `C3 = (C1 통과) AND (반사실 통과)`
- **이 순간 |C3|(C3 통과 개수)가 정해집니다.** C2·C-rand는 이 개수에 맞춰야 하므로 **반드시 먼저** 끝나야 합니다.

**입력:** `unified.jsonl` + `cf_judged.json`
**출력:** `unified.jsonl` (C3 + 반사실 신호 추가됨)

**실행 예시(③ 전체):**
```powershell
py src/filters/counterfactual_adapter.py --total -1 --split train --workers 16 `
   --output data/counterfactual/train_cf.jsonl
py -m filters.counterfactual.precompute --input data/counterfactual/train_cf.jsonl `
   --output data/counterfactual/cf_judged.json --workers 16
py src/dataset/build_arms.py merge-c3 --unified data/unified.jsonl `
   --cf data/counterfactual/cf_judged.json --output data/unified.jsonl
```

### ③보조 `filters/counterfactual/judge.py` — 반사실 채점관의 채점 규칙

**역할:** "이 풀이가 **목표 답(target)** 을 얼마나 강하게 정당화하나"를 1~5로 채점하는 **프롬프트와 파싱 함수**.
**핵심 장치 (label-cheating 방지):** 채점관은 먼저 풀이 본문이 **실제로 어느 선지를 지지하는지**(`supported_letter`)를
스스로 판단합니다. 그게 목표 답과 다르면, 글이 아무리 유창해도 **강제로 1점**. → "마지막 '정답: X' 줄만 보고 속는" 걸 막습니다.

## ④ 범용 judge 필터(C2) — 대조군 ("일반 채점으론 못 잡는다"를 보이기)

### ④-1 `src/filters/judge_general.py` — 풀이 품질을 일반적으로 채점

**역할:** C1 통과 풀이를 **"답을 바꾸지 않고"** 글 자체 품질로 채점합니다. C3와의 **차이를 보여주기 위한 대조군**입니다.

**어떻게 작동하나:**
- 채점관(GPT-5)이 세 축(① justification 근거의 구체성 ② coherence 논리 연결 ③ clarity 명료함)을 각 1~5로 매기고, **평균**을 `c2_score` 로 저장.
- 반사실과 달리 **오답을 만들지 않습니다** — 그냥 "잘 쓴 글인가?"만 봅니다.
- 점수만 박아두므로(판정은 다음 단계), 여러 번 돌려도 추가 비용 없음.

**입력:** `unified.jsonl` (C1 통과 행) + `OPENAI_API_KEY`
**출력:** `unified.jsonl` (`signals.c2_score` 추가)

### ④-merge `build_arms.py merge-c2` — 점수 상위 |C3|개를 C2로

**역할:** `c2_score` 가 높은 순으로 정렬해 **상위 |C3|개**만 `filters.C2 = True`.
- C3와 **똑같은 개수**로 맞춰야 "같은 크기에서 어느 방법이 나은가(H3)"를 공정 비교할 수 있습니다.

**실행 예시:**
```powershell
py src/filters/judge_general.py --unified data/unified.jsonl --output data/unified.jsonl --workers 8
py src/dataset/build_arms.py merge-c2 --unified data/unified.jsonl --output data/unified.jsonl
```

## ⑤ `build_arms.py make-crand` — 수량 통제군(C-rand)

**역할:** C1 통과 풀에서 **무작위로 |C3|개**를 골라 `filters.C-rand = True`.
**왜 필요?** "C3가 좋아진 게 그냥 데이터 양이 줄어서 아니야?"라는 반론을 막기 위함.
→ C-rand는 C3와 **같은 양**이지만 **품질 선별 없이 무작위** → C3 > C-rand 면 "선별이 효과 있다"가 증명됨.

**입력:** `unified.jsonl` (merge-c3 끝난 상태) **출력:** `unified.jsonl` (C-rand 추가)
```powershell
py src/dataset/build_arms.py make-crand --unified data/unified.jsonl --output data/unified.jsonl --seed 42
```

> 여기까지 끝나면 `unified.jsonl` 한 파일에 **5개 묶음 플래그가 전부** 채워집니다.

## ⑥ `src/train/train_lora.py` — 학생 AI 학습 (묶음 × 시드)

**역할:** 각 묶음 데이터로 학생(Qwen3-8B)을 **LoRA 방식**으로 학습시킵니다.

**어떻게 작동하나 (한 묶음당 3단계):**
1. `build_arms export` 를 자동 호출해 묶음별 학습 파일(`data/sft/train_C3.jsonl`) 생성.
2. LLaMA-Factory가 찾을 수 있게 `dataset_info.json` 에 등록.
3. `llamafactory-cli train` 을 [configs/train.yaml](../configs/train.yaml) 설정으로 실행하되,
   **묶음·시드마다 `dataset`/`output_dir`/`seed` 만 바꿔서** 호출.
- 5묶음 × 3시드 = **15번 학습**을 `for arm: for seed:` 이중 루프로 **하나씩 순차** 실행합니다.
- GPU에는 항상 **베이스 모델 + LoRA 1개**만 올라가므로, 15개가 동시에 메모리를 먹지 않습니다.

**왜 시드(seed)를 3개?**
학습엔 무작위성이 있어 한 번 점수는 운에 흔들립니다. 시드 3개로 **평균±표준편차**를 내야
"C3가 C1보다 높다"가 **진짜 효과인지 운인지** 구분됩니다. (⑧ 통계가 이걸 위함)

**입력:** `unified.jsonl` + 학습 설정 + GPU
**출력:** `output/qwen3-8b-<arm>-s<seed>/` (예: `qwen3-8b-c3-s42`) — LoRA 어댑터 폴더

**실행 예시:**
```powershell
py src/train/train_lora.py --unified data/unified.jsonl --arms C0 C1 C2 C3 C-rand --seeds 42 43 44
# 명령만 미리 보기: ... --arms C3 --seeds 42 --dry-run
```

> 참고: [configs/train.yaml](../configs/train.yaml) 이 **유일한 학습 설정**입니다 (LoRA rank 32, 3 epoch, 학습률 1e-4 등).
> 질문 부분은 학습에서 마스킹(=베끼지 않음)하고, **풀이(assistant)만** 학습합니다.

## ⑦ `src/eval/evaluate.py` — 시험 보기 (test 정답률)

**역할:** 학습된 어댑터(또는 베이스 모델)에게 **KorMedMCQA test 문제**를 풀게 해서 정답률을 잽니다.

**어떻게 작동하나:**
- 베이스 Qwen3-8B 를 올리고, `--lora` 로 어댑터를 얹습니다.
- 학습 때와 **똑같은 프롬프트 형식**으로 test 문제를 배치 단위로 풀게 합니다.
- 생성된 답에서 "정답: X" 줄을 파싱해 정답과 비교. 끊겨도 이어서 재개.

**입력:** 베이스 모델 + LoRA 어댑터 폴더 + KorMedMCQA `test`
**출력:** `results/eval_<arm>_s<seed>.jsonl` — 한 줄 = 문항 하나
```json
{ "subset":"doctor", "sample_idx":5, "gold":"B", "predicted":"B", "is_correct":true, "output":"..." }
```

**실행 예시:**
```powershell
py src/eval/evaluate.py --model Qwen/Qwen3-8B --lora output/qwen3-8b-c3-s42 `
   --split test --output results/eval_C3_s42.jsonl
```

> 💡 직접 학습(⑥)을 건너뛰고, huggingface `AIteam4/KoMEDMCQA-LoRA` 에 올라온 어댑터를 받아 바로 ⑦부터 할 수도 있습니다.

## ⑧ `src/eval/stats.py` — 통계 (차이가 진짜인가?)

**역할:** 여러 시드 결과를 모아 **평균±표준편차 / 95% 신뢰구간 / McNemar 검정**을 냅니다.

**어떻게 작동하나:**
- **묶음 내:** 같은 묶음의 시드별 정확도 → 평균·표준편차, 부트스트랩으로 95% 신뢰구간.
- **묶음 간 (McNemar):** 두 묶음이 **같은 문제에서** 누가 맞고 틀렸는지 짝지어 비교 →
  "A만 맞은 수 vs B만 맞은 수"로 p-value 계산 → 단순 평균 비교보다 엄밀하게 "우연인지" 판단.
- 외부 라이브러리 없이 표준 `math`만 사용.

**입력:** `results/eval_*.jsonl` 여러 개 **출력:** `results/stats.json`

**실행 예시:**
```powershell
py src/eval/stats.py `
   --arm C3 results/eval_C3_s*.jsonl --arm C1 results/eval_C1_s*.jsonl `
   --arm C2 results/eval_C2_s*.jsonl --arm C-rand results/eval_C-rand_s*.jsonl `
   --mcnemar C3 C1 --mcnemar C3 C-rand --mcnemar C3 C2 --output results/stats.json
```

## ⑨ `scripts/make_report.py` — 사람이 읽는 리포트

**역할:** 묶음 크기 + 신호 분포 + 정확도 + 검정 결과를 **마크다운 한 장**으로 종합하고 가설마다 ✅/❌를 붙입니다.

**입력:** `unified.jsonl` + `stats.json`
**출력:** `results/report.md` — 다음을 포함:
- **Arm 크기표** (묶음별·과목별 건수)
- **H1 분석 대상** (C1 통과 ∩ C3 탈락 건수 = 일반 청소로는 못 거른 가짜 추론 후보)
- **신호 분포** (gap / orig_score / c2_score 평균·분위수)
- **정확도표 + McNemar** + 가설 점검 (H2/H2통제/H3)

```powershell
py scripts/make_report.py --unified data/unified.jsonl --stats results/stats.json --output results/report.md
```

# 6. 가설 ↔ 어떤 파일이 증명하나

| 가설 | 의미 | 비교 | 관련 단계·파일 |
|---|---|---|---|
| **H1** | 반사실은 일반 청소가 **못 잡는** 별개 문제를 잡는다 | C1 통과 ∩ C3 탈락 건수 | ⑨ make_report (학습 불필요) |
| **H2** | 제안 필터가 성능을 높인다 | C3 > C1, C3 > C-rand | ⑥~⑧ (mcnemar C3 C1 / C3 C-rand) |
| **H3** | 일반 judge로는 못 잡는 걸 반사실이 잡는다 | C3 > C2 (같은 크기) | ④→⑥~⑧ (mcnemar C3 C2) ← **가장 중요** |

# 7. 가장 헷갈리는 3가지 — 짧게 정리

1. **judge가 두 개다.**
   - C2 judge([judge_general.py](../src/filters/judge_general.py)): 답 안 바꾸고 "글 잘 썼나?" → 대조군.
   - C3 judge([judge.py](../filters/counterfactual/judge.py)): 답을 오답으로 바꿔 "그래도 정당화되나?" → 제안 방법.

2. **순서가 중요하다.** C3가 **|C3| 개수를 먼저 확정**해야(merge-c3), C2(상위 |C3|개)와 C-rand(무작위 |C3|개)를 만들 수 있습니다.

3. **비싼 단계와 싼 단계가 분리돼 있다.** 반사실은 `precompute`(생성+채점, 비쌈)에서 점수를 다 박아두고,
   `run`(판정, 산수만)은 임계값을 바꿔 몇 번이고 무료로 다시 돌립니다.
